### 🐛 Bug de layout — diagnóstico

#### Desktop: SIN BUG CONFIRMADO
En `00-landing-hero` el contenido aparece correctamente above the fold. El tag, título, CTAs y mockup son visibles sin scroll. El fix del `lp-hero` que eliminó el `min-height: calc(100dvh - 53px)` + `align-items: center` está aplicado correctamente.

#### Mobile iOS: BUG CONFIRMADO — Gap negro entre features y módulos

**Evidencia**: En `responsive-ios-landing` hay un bloque negro vacío de ~300px entre las cards de "World Map / Plan Aliados" y la siguiente sección de módulos (que empieza con el card de Productividad).

**Diagnóstico de causa raíz**: La estructura probable es:
```html
<section class="lp-features">     <!-- lista World Map, Plan Aliados... -->
  <div class="lp-features-list">...</div>
</section>
<div class="lp-hero-visual">      <!-- mockup del teléfono, visible en desktop -->
  ...
</div>
<section class="lp-modules">     <!-- cards de módulos -->
```

El `lp-hero-visual` (el mockup del teléfono que aparece a la derecha en desktop como `position: absolute` o como columna del grid) en mobile probablemente sigue reservando altura pero el contenido está oculto con `display:none` o `visibility:hidden` — manteniendo el espacio en el documento.

**Fix específico**:
```css
/* En el breakpoint mobile */
@media (max-width: 768px) {
  /* Si el visual del hero es un elemento del flex/grid que se oculta */
  .lp-hero-visual,
  .lp-hero-mockup,
  .lp-phone-mockup {
    display: none; /* ya existe */
    height: 0;     /* AGREGAR ESTO */
    overflow: hidden;
    margin: 0;
    padding: 0;
  }
  
  /* Alternativamente si es un contenedor grid */
  .lp-hero {
    grid-template-columns: 1fr; /* colapsar a 1 col */
  }
  
  /* El gap entre secciones */
  .lp-features-section {
    margin-bottom: 0;
    padding-bottom: 24px;
  }
  
  .lp-modules-section {
    margin-top: 24px;
  }
}
```

Si el elemento es el contenedor del mockup flotante con `position: relative` y `min-height` heredada:
```css
@media (max-width: 768px) {
  .lp-hero-right,
  .lp-visual-col {
    position: static !important;
```

### 🎨 Animación de colores glitch — evaluación + implementación

#### Evaluación de la propuesta

**¿Beneficia o perjudica la conversión?**

**Veredicto: BENEFICIA — con condiciones de ritmo.**

El título es lo primero que lee el usuario. Un gradiente que cicla entre los 8 colores preset comunica visualmente el concepto de "personalización" sin necesitar explicarlo — el usuario ve el color cambiar y entiende intuitivamente que "eso puede ser *su* color". Para un producto cuyo core differentiator es la identidad visual personalizable, este movimiento tiene significado semántico, no es decoración.

El efecto glitch refuerza la estética gaming/cyberpunk del producto. No es un glitch gratuito — es coherente con "Juega en serio."

**Riesgos a mitigar**:
1. Si el ciclo es muy rápido (<2s) → mareo, distracción del copy
2. Si el glitch es muy largo (>0.3s) → parece roto, no estético
3. Si no hay `prefers-reduced-motion` → accesibilidad fallida

**Ritmo recomendado**: Cada color dura **3.5 segundos** en estado estable → **0.15s de glitch** → siguiente color. Ciclo completo: ~30 segundos para los 8 colores. Esto es suficiente para que el usuario lo note si mira 5+ segundos (tiempo promedio en hero de landing), pero no tan rápido que distraiga.
